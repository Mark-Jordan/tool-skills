# 模型选择依据

## 默认方案

默认使用 `faster-whisper` 的 CPU int8 路线。

理由：

- `faster-whisper` 基于 CTranslate2，适合 CPU int8 推理。
- 相比原版 OpenAI Whisper，CPU 推理速度和内存占用更适合本地批处理。
- 当前脚本已经围绕 `Systran/faster-whisper-*` 模型缓存实现，改动成本低。

推荐顺序：

1. `base`：默认值，速度和准确率平衡。
2. `small`：CPU 可接受时优先用于正式字幕。
3. `medium`：准确率更高，但 CPU 耗时明显增加。
4. `tiny`：仅用于快速预检或低资源机器。

## 备选方案

### whisper.cpp

适合追求极简部署、无 Python 运行时或嵌入式场景。若当前 skill 继续依赖 Python、文档读取和后处理，迁移收益有限。

### OpenAI Whisper 原版

模型质量稳定，但 CPU 推理通常不如 `faster-whisper` 适合批处理。除非需要与原版输出严格对齐，否则不作为默认方案。

### 硬字幕 OCR

适合视频画面已存在字幕、音频噪声大或无音轨的场景。它依赖抽帧、字幕区域检测和 OCR，工程链路与音频 ASR 不同，建议作为后续独立分支实现。

## 决策规则

- 用户没有特别指定时，使用音频 ASR。
- 用户明确说明“画面上已有字幕”“提取视频上显示的字幕”“音频不清楚”时，提示 OCR 路线更合适。
- 只用 CPU 时，不默认推荐大型端到端多模态模型；优先选择可缓存、可离线、可解释失败原因的 ASR/OCR 流水线。
